# `abcvlib` Comprehensive Testing Strategy & Proposal

This document outlines the testing strategy, architectural evaluation, and complete test-suite plan for the `abcvlib` library and its associated demo applications. Due to the high-frequency, multi-threaded nature of cyber-physical smartphone-robot interactions, a robust multi-tiered testing plan is essential to prevent hardware regressions, memory leaks, and communication breakdowns.

---

## 1. Architecture Overview & Testing Vectors

The `abcvlib` library coordinates real-time telemetry from multiple phone/microcontroller sensors, processes inputs using on-device machine learning (MediaPipe / TFLite) and signal processing, serializes telemetry into FlatBuffer episodes, and communicates with:
1. **The RP2040 Microcontroller** over high-speed USB-CDC serial.
2. **A Remote Learning Server** over raw TCP sockets.

To ensure comprehensive test coverage, we divide the test suite into four complementary tiers:

```
┌───────────────────────────────────────────────────────────────────────────┐
│                          4. UI & SMOKE TESTS (End-to-End)                 │
│         - ComprehensiveDemo Smoke   - PIDBalancer Loop   - HandsOnUI      │
└─────────────────────────────────────┬─────────────────────────────────────┘
                                      ▼
┌───────────────────────────────────────────────────────────────────────────┐
│                    3. EMULATED HARDWARE & PROTOCOL TESTS                  │
│         - Mock USB-CDC Serial       - Custom Packet Validation            │
└─────────────────────────────────────┬─────────────────────────────────────┘
                                      ▼
┌───────────────────────────────────────────────────────────────────────────┐
│                    2. INSTRUMENTED INTEGRATION TESTS (Device)             │
│         - Multi-phase Publisher    - CameraX & ML Model   - Audio/IMU     │
└─────────────────────────────────────┬─────────────────────────────────────┘
                                      ▼
┌───────────────────────────────────────────────────────────────────────────┐
│                        1. LOCAL UNIT TESTS (JVM / Host)                   │
│         - DSP & Image Compression  - FlatBuffer Serialization             │
│         - File & Storage Operators  - TimeStepBuffer Lock Synchronization │
└───────────────────────────────────────────────────────────────────────────┘
```

## 2. Test Categories and Proposed Cases

### 2.1 Category 1: Local Unit Tests (Local JVM Tests)
**Target:** Fast, deterministic verification of core mathematical computations, utility algorithms, and serialization models running on the developer's machine without an emulator.

#### Proposed Test Cases:
1. **`DSPTest` (Mathematics & Smoothing)**
   - *Test Case 1.1:* Verify that `exponentialAvg` calculates the correct output based on weighting factors ($w \in [0.0, 1.0]$).
   - *Test Case 1.2:* Validate edge-case weightings ($0.0$ and $1.0$) to guarantee output boundaries are never exceeded.
2. **`HexBinConvertersTest` (Protocol Serialization)**
   - *Test Case 2.1:* Ensure binary streams/bytes are correctly converted into accurate hexadecimal string representations and vice versa.
   - *Test Case 2.2:* Check robust handling of boundary conditions (empty arrays, negative byte values, null characters).
3. **`ImageOpsTest` (Image Processing)**
   - *Test Case 3.1:* Verify that `compressImage` successfully reduces the size/quality of a bitmap without corrupting the pixel data.
   - *Test Case 3.2:* Validate that `generateBitmap` and `compressImage` create a valid round-trip (byte array to Bitmap and back) without data loss.
   - *Test Case 3.3:* Ensure `addCompressedImage2Buffer` correctly compresses a bitmap into a WebP byte array and assigns it to the target `TimeStepData` buffer index.
   *Note: Requires Robolectric for local JVM execution of `android.graphics` classes.*
4. **`FlatbufferAssemblerTest` (Serialization Pipeline)**
   - *Test Case 3.1:* Verify `FlatbufferAssembler` accurately serializes various nested telemetry inputs (`WheelData`, `ChargerData`, `OrientationData`, `Audio`, compressed WebP images) into an `Episode` flatbuffer matching the schema structure.
   - *Test Case 3.2:* De-serialize the generated byte buffer using FlatBuffer readers and assert that fields match original inputs perfectly.
   - *Test Case 3.3:* Test empty state or missing sensors (e.g. sound data empty) to verify the builder doesn't throw a `NullPointerException`.
5. **`TimeStepDataBufferTest` (Data Structures & Synchronization)**
   - *Test Case 4.1:* Verify thread safety of the synchronized circular cache. Assert that calling `nextTimeStep()` correctly loops the write pointer, updates the read pointer, and cleans up the overwritten timestep index.
   - *Test Case 4.2:* Verify that `TimeStepDataBuffer` throws `IllegalArgumentException` if initialized with a buffer length $\le 1$.
   - *Test Case 4.3:* Validate background image WebP compression futures tracking; ensure that `imgCompFuturesTimeStep` collects all background tasks and executes them asynchronously.
6. **`AndroidToRP2040PacketTest` (Protocol Serialization)**
   - *Test Case 5.1:* Verify that the packet structure maintains the correct total size and correctly places the start (0x02) and stop (0x03) byte markers.
   - *Test Case 5.2:* Assert that `setCommand` correctly places the command hex value in the second byte of the packet.
   - *Test Case 5.3:* Validate that the `clear()` method resets the internal buffer states to allow for repeated, consistent packet generation.
---

### 2.2 Category 2: Instrumented Integration Tests (Device/Emulator)
**Target:** Validate hardware sensors (IMU, CameraX, Microphone), multi-phase registration, and Android-specific concurrency structures.

#### Proposed Test Cases:
1. **`PublisherManagerConcurrencyTest` (Phased Initialization)**
   - *Test Case 1.1:* Verify `PublisherManager`'s 3-phase synchronization barrier using a Mock Publisher.
   - *Test Case 1.2:* Ensure Phase 0 (Permissions) blocks initialization until permissions are registered as granted.
   - *Test Case 1.3:* Verify that Phase 1 (Thread Init via `Phaser`) and Phase 2 (Resume/Record) function correctly under high concurrent publisher volumes.
2. **`CameraXAndObjectDetectorIntegrationTest` (Vision & ML Processing)**
   - *Test Case 2.1:* Load mock vision frames into `ObjectDetectorData` with the standard model (`efficientdet-lite1.tflite`).
   - *Test Case 2.2:* Assert that image frames are correctly parsed and target classification tags/bounding boxes are updated inside subscribers.
   - *Test Case 2.3:* Check `YuvToRgbConverter` conversion performance and output bitmap resolution consistency.
3. **`MicrophoneDataCollectionTest` (Audio Metrics)**
   - *Test Case 3.1:* Verify real-time raw PCM data sampling rates, audio duration Calculations, and amplitude level calculations are correctly stored in `TimeStepData.SoundData`.
4. **`SensorDataStructuresTest` (Telemetry Containers)**
   - *Test Case 4.1:* Verify that `BatteryData` correctly stores and retrieves multiple timestamped voltage entries, maintaining array order and size integrity.
   - *Test Case 4.2:* Verify that `WheelData` maintains separate, correct instances for `Left` and `Right` wheel data, ensuring that values put into the left channel do not bleed into the right.
   - *Test Case 4.3:* Validate that the array-based retrieval methods for `WheelData` (e.g., `getTimeStamps()`, `getCounts()`, `getSpeedsInstantaneous()`) accurately return the arrays populated by `put()` calls.
5. **`ImageDataContainerTest`**
   - *Test Case 5.1:* Verify that `ImageData` correctly maps timestamps to `SingleImage` objects (Bitmap + WebP).
   - *Test Case 5.2:* Assert that the `ImageData` container correctly handles image replacement or cleanup when new frames arrive in the circular buffer.
6. **`OrientationDataCollectionTest`**
   - *Test Case 6.1:* Verify that tilt angle (radians) and angular velocity (rad/s) are correctly stored with high-precision `System.nanoTime()` timestamps.
   - *Test Case 6.2:* Assert that `getTiltAngle()` and `getAngularVelocity()` return arrays of consistent size with the input `put()` calls.
7. **`QRCodeIntegrationTest`**
   - *Test Case 7.1:* Feed synthetic bitmap frames containing valid QR patterns to the `QRCodeDataSubscriber` pipeline.
   - *Test Case 7.2:* Assert that the detector correctly triggers `onQRCodeDetected` with the expected decoded string.
8. **`LearningOrchestrationTest` (Trial & State Management)**
   - *Test Case 8.1:* Verify that `StateSpace` correctly registers with the `PublisherManager` and that `ActionSpace` correctly wraps the `Comm` and `Motion` action sets.
   - *Test Case 8.2:* Verify that `Trial` correctly initializes the state machine and invokes the `abcvlibMainLoop` in the expected execution order when initialized with the `PublisherManager`.
   - *Test Case 8.3:* Assert that `endEpisode()` correctly triggers a reset of the `TimeStepDataBuffer` write pointer and clears any pending compression futures.
9. **`FileOpsTest` (Storage Persistence)**
   - *Test Case 9.1:* Verify that `savedata(context, ...)` correctly persists bytes to the internal application storage directory and can be read back.
   - *Test Case 9.2:* Assert that `writeToFile` and `readData` correctly preserve precision when serializing/deserializing `DoubleArray` telemetry data to the file system.
   - *Test Case 9.3:* Verify that the storage utility correctly handles missing files without crashing and checks storage availability status appropriately.
10. **`HttpConnectionTest` (Network & IO)**
   - *Test Case 10.1:* Verify that `sendData` correctly transmits POST requests with the expected custom headers for different `HttpDataType` configurations and triggers `onSuccess`.
   - *Test Case 10.2:* Verify `getData` correctly differentiates between file downloads (saving to `getExternalFilesDir`) and in-memory responses based on `Content-Disposition` headers.
   - *Test Case 10.3:* Validate error handling by simulating server-side failures (404/500) and asserting that the `onError` callback is triggered with the parsed error stream content.

### 2.3 Category 3: Emulated Hardware & Protocol Tests
**Target:** Mimic the communication between the phone and the RP2040 microcontroller. This fulfills the **`docs/milestones/InstrumentationTests.md`** requirement to allow fully-automated, remote testing on CI systems without a physical robot.

```
                  ┌────────────────────────────────────────┐
                  │       MainActivity / Controller        │
                  └───────────────────┬────────────────────┘
                                      │ (Outputs)
                                      ▼
                  ┌────────────────────────────────────────┐
                  │          SerialCommManager             │
                  └───────────────────┬────────────────────┘
                                      │ (Writes Packet)
                                      ▼
                  ┌────────────────────────────────────────┐
                  │             MockUsbSerial              │
                  │   - Simulates Serial Port IO           │
                  │   - Feeds Back Fake RP2040Packets      │
                  └────────────────────────────────────────┘
```

#### Proposed Test Cases:
1. **`SerialCommManagerProtocolTest`**
   - *Test Case 1.1:* Verify that executing `outputs.setWheelOutput()` translates into an `AndroidToRP2040Packet` containing correct PWM limits, brakes, and commands.
   - *Test Case 1.2:* Verify that `SerialCommManager` defaults to writing `GET_STATE` commands periodically every 10ms when idle.
2. **`HardwareEmulationLoopTest`**
   - *Test Case 2.1:* Configure `MockUsbSerial` to bypass physical serial connections and feed back pre-packaged `RP2040ToAndroidPacket` byte frames (containing fake encoder counts and voltage levels).
   - *Test Case 2.2:* Assert that the status loop registers the simulated packet, parses states, and updates `BatteryData` and `WheelData` subscribers seamlessly.
   - *Test Case 2.3:* Simulate connection degradation (loss of packets, bad stop/start markers, or connection resets) and verify that the library triggers the exception handler without hard crashing the main thread.

### 2.4 Category 4: UI & Smoke Tests for Demo Applications
**Target:** Validate that demo applications compile, hook up `abcvlib` lifecycles correctly, grant runtime permissions programmatically, and execute control loops without regressions.

#### Proposed Test Cases:
1. **`MainActivityHardwareSmokeTest` (ComprehensiveDemo)**
   - *Test Case 1.1 (Existing):* Use UI Automator to launch `ComprehensiveDemo`'s `MainActivity`, handle system permission dialogues, and verify that the controller cycles through all behaviors (`Behavior.entries`) without crashing.
2. **`HandsOnAppControlLoopTest` (HandsOnApp)**
   - *Test Case 2.1:* Verify `handsOnApp`'s `MainActivity` registers object detection subscribers, updates screen indicators on state transitions, and stops wheel motor output when target labels are detected.
3. **`PIDBalancerControlLoopTest` (PIDBalancer)**
   - *Test Case 3.1:* Feed simulated IMU angle deviations (simulating tilt offsets) and assert that the internal `BalancePIDController` computes proportional-integral-derivative responses, driving wheel motor outputs back toward equilibrium.
4. **`PidGuiFragmentSmokeTest`**
   - *Test Case 4.1:* Instantiate `PidGuiFragment` with a mock `BalancePIDController` and verify that layout widgets correctly inflate.
   - *Test Case 4.2:* Programmatically adjust the UI sliders and verify that `balancePIDController.setPID()` is invoked with the expected parameters mapped from the UI.
5. **`QRCodeDisplayFragmentTest`**
   - *Test Case 5.1:* Launch `QRCodeDisplayFragment` with a known test string and verify that the `ImageView` is populated with a valid, non-null `Bitmap`.
   - *Test Case 5.2:* Assert that the encoded bitmap dimensions conform to the parent container's layout constraints.

### 3.1 Local Unit Test: `FlatbufferAssemblerTest`
*File path suggestion:* `libs/abcvlib/src/test/java/jp/oist/abcvlib/core/learning/FlatbufferAssemblerTest.kt`

```kotlin
package jp.oist.abcvlib.core.learning

import jp.oist.abcvlib.core.inputs.TimeStepDataBuffer
import jp.oist.abcvlib.core.learning.fbclasses.Episode
import org.junit.Assert.assertEquals
import org.junit.Assert.assertNotNull
import org.junit.Test
import java.net.InetSocketAddress
import java.nio.ByteBuffer

class FlatbufferAssemblerTest {

    @Test
    fun testFlatbufferSerializationAndStructure() {
        val maxSteps = 10
        val trial = Trial(maxSteps)
        val dataBuffer = TimeStepDataBuffer(50)
        
        // Populate standard dummy telemetry metrics
        dataBuffer.onBatteryVoltageUpdate(System.nanoTime(), 12.6)
        dataBuffer.onOrientationUpdate(System.nanoTime(), 0.15, -0.02)
        dataBuffer.onWheelDataUpdate(
            System.nanoTime(),
            100, 105,
            1.2, 1.25,
            0.5, 0.52,
            0.5, 0.51,
            0.5, 0.51
        )
        
        val assembler = FlatbufferAssembler(
            myTrial = trial,
            inetSocketAddress = InetSocketAddress("127.0.0.1", 8080),
            socketListener = null,
            timeStepDataBuffer = dataBuffer,
            robotID = 42
        )

        assembler.startEpisode()
        assembler.addTimeStep(0)
        
        // Wait for all asynchronous flatbuffer threads to finish
        for (future in assembler.flatbufferWriteFutures) {
            future.get()
        }
        
        assembler.endEpisode()
        
        // Grab output serialized buffer
        val field = FlatbufferAssembler::class.java.getDeclaredField("episode")
        field.isAccessible = true
        val buffer = field.get(assembler) as ByteBuffer
        
        assertNotNull("Serialized episode buffer should not be null", buffer)
        
        // Deserialize and check field mappings
        val episodeObj = Episode.getRootAsEpisode(buffer)
        assertEquals(42, episodeObj.robotid())
        assertNotNull(episodeObj.timesteps(0))
    }
}
```

### 3.2 Instrumented Integration Test: `PublisherManagerConcurrencyTest`
*File path suggestion:* `libs/abcvlib/src/androidTest/java/jp/oist/abcvlib/core/inputs/PublisherManagerConcurrencyTest.kt`

```kotlin
package jp.oist.abcvlib.core.inputs

import android.content.Context
import androidx.test.core.app.ApplicationProvider
import androidx.test.ext.junit.runners.AndroidJUnit4
import org.junit.Assert.assertEquals
import org.junit.Test
import org.junit.runner.RunWith
import java.util.concurrent.CountDownLatch
import java.util.concurrent.TimeUnit

@RunWith(AndroidJUnit4::class)
class PublisherManagerConcurrencyTest {

    private class MockPublisher(context: Context, manager: PublisherManager) :
        Publisher<Subscriber>(context, manager) {
        
        var startedCalled = false
        var stoppedCalled = false

        override fun getRequiredPermissions(): ArrayList<String> {
            return arrayListOf() // No dangerous permissions needed for mock tests
        }

        override fun start() {
            super.start()
            startedCalled = true
            publisherManager.onPublisherInitialized()
        }

        override fun stop() {
            super.stop()
            stoppedCalled = true
        }
    }

    @Test
    fun testPublisherManagerPhasedLifecycle() {
        val context = ApplicationProvider.getApplicationContext<Context>()
        val manager = PublisherManager()
        val publisher = MockPublisher(context, manager)

        assertEquals(PublisherState.STOPPED, publisher.getState())

        // Initializing transition to Phase 1 (Arrive & Await Phase 0 completion)
        manager.initializePublishers()
        
        // Verify that publisher.start was invoked during the phase progression
        assertEquals(true, publisher.startedCalled)

        // Start publishers (Phase 2 progression)
        manager.startPublishers()
        
        // Give asynchronous ExecutorService some moments to resume
        Thread.sleep(500)
        assertEquals(PublisherState.STARTED, publisher.getState())

        // Shutdown and check STOP state transition
        manager.stopPublishers()
        assertEquals(true, publisher.stoppedCalled)
        assertEquals(PublisherState.STOPPED, publisher.getState())
    }
}
```

### 3.3 Hardware Mock Integration Test: `SerialCommManagerTest`
*File path suggestion:* `libs/abcvlib/src/androidTest/java/jp/oist/abcvlib/util/SerialCommManagerTest.kt`

```kotlin
package jp.oist.abcvlib.util

import android.content.Context
import android.hardware.usb.UsbManager
import androidx.test.core.app.ApplicationProvider
import androidx.test.ext.junit.runners.AndroidJUnit4
import jp.oist.abcvlib.core.inputs.microcontroller.BatteryData
import jp.oist.abcvlib.core.inputs.microcontroller.WheelData
import jp.oist.abcvlib.core.inputs.PublisherManager
import org.junit.Assert.assertEquals
import org.junit.Assert.assertNotNull
import org.junit.Before
import org.junit.Test
import org.junit.runner.RunWith
import org.mockito.Mockito.*
import java.io.IOException
import java.nio.ByteBuffer
import java.nio.ByteOrder

@RunWith(AndroidJUnit4::class)
class SerialCommManagerTest {

    private lateinit var mockUsbSerial: UsbSerial
    private lateinit var batteryData: BatteryData
    private lateinit var wheelData: WheelData
    private lateinit var publisherManager: PublisherManager
    private lateinit var serialCommManager: SerialCommManager

    @Before
    fun setUp() {
        val context = ApplicationProvider.getApplicationContext<Context>()
        publisherManager = PublisherManager()
        batteryData = BatteryData.Builder(context, publisherManager).build()
        wheelData = WheelData.Builder(context, publisherManager).build()
        
        // Mock physical UsbSerial driver
        mockUsbSerial = mock(UsbSerial::class.java)
        serialCommManager = SerialCommManager(mockUsbSerial, batteryData, wheelData)
    }

    @Test
    fun testSerialCommManagerPeriodicGetStateFormulation() {
        // Access non-public periodic byte generation
        val method = SerialCommManager::class.java.getDeclaredMethod("generateGetStateCmd")
        method.isAccessible = true
        val commandBytes = method.invoke(serialCommManager) as ByteArray

        assertNotNull(commandBytes)
        assertEquals(AndroidToRP2040Packet.packetSize, commandBytes.size)
        
        // Verify start and end packet markers
        assertEquals(0x02.toByte(), commandBytes[0]) // START marker
        assertEquals(AndroidToRP2040Command.GET_STATE.value.toByte(), commandBytes[1])
        assertEquals(0x03.toByte(), commandBytes[commandBytes.size - 1]) // STOP marker
    }

    @Test
    fun testReceivePacketStateIntegration() {
        // Simulate physical incoming RP2040ToAndroidPacket data frame
        val buffer = ByteBuffer.allocate(64).apply {
            order(ByteOrder.LITTLE_ENDIAN)
            put(0x02.toByte()) // START marker
            put(AndroidToRP2040Command.GET_STATE.value.toByte()) // CMD
            putDouble(12.4) // Voltage Level
            putInt(500) // Left encoder count
            putInt(512) // Right encoder count
            // ... padding rest with zero
            position(63)
            put(0x03.toByte()) // STOP marker
        }
        val packetBytes = buffer.array()

        // Verify simulated state translation
        val parseStatusMethod = SerialCommManager::class.java.getDeclaredMethod("parseStatus", ByteArray::class.java)
        parseStatusMethod.isAccessible = true
        parseStatusMethod.invoke(serialCommManager, packetBytes)

        // Assert updates propagated cleanly down to local subscriber data stores
        // Note: Check state references parsed from RP2040State
        val stateField = SerialCommManager::class.java.getDeclaredField("rp2040State")
        stateField.isAccessible = true
        val rp2040State = stateField.get(serialCommManager) as RP2040State
        
        assertNotNull(rp2040State)
    }
}
```

### 3.4 UI & Controller Loop Test: `PIDBalancerControlLoopTest`
*File path suggestion:* `apps/pidBalancer/src/androidTest/java/jp/oist/abcvlib/pidbalancer/PIDBalancerControlLoopTest.kt`

```kotlin
package jp.oist.abcvlib.pidbalancer

import androidx.test.ext.junit.runners.AndroidJUnit4
import jp.oist.abcvlib.tests.BalancePIDController
import org.junit.Assert.assertTrue
import org.junit.Test
import org.junit.runner.RunWith

@RunWith(AndroidJUnit4::class)
class PIDBalancerControlLoopTest {

    @Test
    fun testPIDControllerOutputOnPitchDeviation() {
        val controller = BalancePIDController()
        
        // Access controller parameters and simulate high deviation values
        val thetaField = BalancePIDController::class.java.getDeclaredField("thetaDeg")
        thetaField.isAccessible = true
        
        // Set a high forward tilt angle deviation (setpoint is ~2.8)
        thetaField.set(controller, 15.0)

        // Trigger control loop cycle
        controller.run()

        // Access internal speed variables to verify counteractive wheel adjustments
        val speedLField = BalancePIDController::class.java.getDeclaredField("speedL")
        speedLField.isAccessible = true
        val computedSpeedL = speedLField.get(controller) as Double

        // Assert robot computes negative/backwards speed output to counteract the tilt
        assertTrue("Calculated speed output must adjust to balance high tilt deviation", computedSpeedL != 0.0)
    }
}
```

## 4. How to Run the Proposed Test Suite

### 4.1 Via Android Studio
1. Right-click the test folder or class file in the project explorer tree.
2. Select **Run 'FlatbufferAssemblerTest'** or **Run 'All Tests'**.
3. Choose to target either the host computer JVM (for Unit Tests) or a running Android Emulator/Device (for Instrumented and Integration tests).

### 4.2 Via Gradle CLI
Open terminal inside the project root folder and execute:

```bash
# Execute all local JVM unit tests inside the abcvlib module
./gradlew :abcvlib:testDebugUnitTest

# Execute instrumented integration and mock hardware tests on target emulator/device
./gradlew :abcvlib:connectedDebugAndroidTest

# Run end-to-end user-experience smoke tests inside demo apps (e.g. comprehensiveDemo)
./gradlew :comprehensiveDemo:connectedDebugAndroidTest
```
